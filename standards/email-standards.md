# Rock Agile Email Standards

This document defines the voice, formatting rules, and HTML template for Rock Agile client-facing emails.

Last updated: 2026-05-26 (signature system, heading-color refinement, bold sub-topics, em-dash hardening)

---

## 1. Voice and Tone

All Rock Agile emails must reflect the brand voice:

- **Direct** - say what we mean, no filler, no buzzwords
- **Confident but not arrogant** - we make specific claims plainly because we've earned them
- **Honest** - including about tradeoffs, limits, and what we're not the right fit for
- **Warm and human** - we're a firm, but we're people; long relationships are part of who we are

### Banned words and phrases

> **EM-DASHES ARE BANNED.** Never use `—` in drafted content. Use a regular hyphen with spaces (` - `) or rewrite the sentence. This applies to every word of generated email content including the signature. The `/ra-email` command MUST scan output for `—` before creating the draft and fix any it finds.

**Never use:** "synergize," "leverage," "empower," "streamline," or "solutions" without something concrete behind the word.

**Never use these AI-tell words/phrases:** "utilize" (use "use"), "delve," "nuanced," "ensure," "tapestry," or "ecosystem"/"landscape" as metaphors.

**Never use these filler openers:** "Certainly," "Absolutely," "Great question," "Of course," "I'd be happy to," "I hope this finds you well."

**Never use these transitional fillers:** "That being said," "With that in mind," "It's worth noting that," "Furthermore," "Moreover."

**Never use rhetorical flourishes of the form** "the X it deserves/warrants/requires/needs" (e.g. "the focus it deserves," "the attention it warrants"). These are AI tells - just state the thing plainly.

**Never make claims we can't back up** with a number or a story.

## 2. Design Principles

- **No tables in the email body.** Tables render inconsistently across email clients and are fragile to maintain. Use only block-level elements (`<p>`, `<div>`, `<hr>`) with inline styles. **Exception:** the signature block uses a `<table>` because the image-divider-text layout is the cross-client-safe convention for signatures. The signature is the only place a table is allowed.
- **Outlook compatibility.** Many Rock Agile clients use Outlook, which renders HTML using Word's engine. Every element in this template must render correctly in both Outlook and modern email clients (Gmail, Apple Mail, etc.). When a CSS property is unreliable in Outlook, use the equivalent HTML attribute alongside it.
- **Inline styles only.** Gmail strips `<style>` blocks. All styles must be inline on the element.

## 3. Style Rules

### Structure

Emails follow this pattern (red `<hr>` rules separate each chunk):

1. **Top brand accent** - red `<hr>` at the very top of the email
2. **Opening line(s)** - one or two sentences of context or greeting (no heading)
3. **Red `<hr>`** - section divider
4. **Section 1 heading** + content (see "Heading colors" below for the color)
5. **Red `<hr>`** - section divider
6. **Section 2 heading** + content
7. (repeat 5-6 for each additional major section)
8. **Red `<hr>`** - section divider before the closing
9. **Closing line** - clear next step or call to action. No "please don't hesitate to reach out." No heading.
10. **Signature** - generated by `/ra-email` from the hosted template plus the user's local config. See Section 5. No `<hr>` between closing and signature (the signature has its own visual structure).

Other formatting:

- **Lists** - dash-prefixed lines for unordered, or `<ol>` for true step sequences where order matters. Avoid `<ul>/<li>` (inconsistent across clients).
- **Bold inline sub-topic labels** - short labels like `Downtime window:` or `Pre-flight checks - ` start a paragraph or list item, in **bold** (not italic). Italic remains acceptable for longer secondary headers where bold would feel heavy, but bold is the default.

### Heading colors

The heading-color rule is **importance-ranked**, not keyword-driven. Pick the most important section heading, color it red; pick the second most important, color it blue; everything else is black. Mechanical, no judgment about whether a section "is urgent enough" to deserve red.

- **Most important section heading: RED (#EF423C).** Whatever is THE primary topic of the email - the plan, the proposal, the answer, the ask. Exactly one red heading per email.
- **Second most important section heading: BLUE (#5481C9).** When the email has at least two major sections, the secondary one gets blue. Common pattern: red = "what we're doing" / blue = "what to expect" / "context" / "supporting info."
- **All other section headings: BLACK (#231F20).** Tertiary, administrative, FYI sections.

**Mechanical cases:**

- Single-section email: that one heading is RED (it IS the most important by definition).
- Two-section email: RED + BLUE.
- Three-or-more-section email: RED + BLUE + BLACK(s).
- No-heading email (short note with no major sections): no colored headings needed.

**Edge case - genuinely co-equal sections:** if two sections are honestly the same importance, default to RED for whichever appears first, BLUE for the second. Don't try to leave one without color. The hierarchy is the visual cue; missing the cue makes the email read flat.

### Formatting Details

- **Top accent line** - a 3px Rock Agile Red `<hr>` at the top of the email container. Present on ALL emails (with or without section headings) for consistent brand presence.
- **Headings** are bold text on their own line, same font size as body text - the weight distinguishes them. Color per the "Heading colors" rule above.
- **Body text** is 15px, dark gray (#231F20), with comfortable line-height (1.6).
- **Lists** use `- ` prefix for unordered, or `<ol>` for ordered. Each item gets its own line. No trailing periods unless the item is a full sentence.
- **Bold sub-topic labels** are short bold labels that start a paragraph or list item, typically followed by `:` or ` - ` and the description on the same line.
- **Links** use Rock Agile Blue (#5481C9).
- **Font stack**: `Arial, sans-serif` - matches Gmail's default "Sans Serif" font. Email clients don't reliably load custom fonts (Montserrat included), so we use the same sans-serif recipients see by default. The brand font (Montserrat) is reserved for documents and print where it can actually load.

## 4. HTML Template

```html
<div style="font-family:Arial,sans-serif;font-size:15px;color:#231F20;line-height:1.6;max-width:680px;">

  <!-- Top brand accent line -->
  <hr style="border:none;border-top:3px solid #EF423C;margin:0 0 24px 0;" color="#EF423C" size="3" noshade>

  <!-- Opening paragraph -->
  <p style="margin:0 0 16px 0;">
    {{opening}}
  </p>

  <!-- Section divider -->
  <hr style="border:none;border-top:3px solid #EF423C;margin:24px 0;" color="#EF423C" size="3" noshade>

  <!-- Section 1 heading (most important = RED) -->
  <p style="margin:24px 0 8px 0;font-weight:bold;color:#EF423C;">
    {{most-important section heading}}
  </p>
  <div style="margin:0 0 16px 0;">
    {{section content}}
  </div>

  <!-- Bold sub-topic label inline at the start of a paragraph (used inside section content) -->
  <p style="margin:0 0 16px 0;">
    <strong>{{sub-topic label:}}</strong> {{description on the same line}}
  </p>

  <!-- Section divider -->
  <hr style="border:none;border-top:3px solid #EF423C;margin:24px 0;" color="#EF423C" size="3" noshade>

  <!-- Section 2 heading (second most important = BLUE) -->
  <p style="margin:24px 0 8px 0;font-weight:bold;color:#5481C9;">
    {{second-most-important section heading}}
  </p>
  <div style="margin:0 0 16px 0;">
    {{section content}}
  </div>

  <!-- Section divider before closing -->
  <hr style="border:none;border-top:3px solid #EF423C;margin:24px 0;" color="#EF423C" size="3" noshade>

  <!-- Closing paragraph (no heading) -->
  <p style="margin:24px 0 0 0;">
    {{closing}}
  </p>

  <!-- Signature is appended after this block by /ra-email; no <hr> between closing and signature -->

</div>
```

For a third (or further) section heading: BLACK (`color:#231F20;`).

### Headerless Emails

For short emails with no section headings (quick replies, single-topic messages), the template simplifies to just the accent line, body, and closing:

```html
<div style="font-family:Arial,sans-serif;font-size:15px;color:#231F20;line-height:1.6;max-width:680px;">

  <hr style="border:none;border-top:3px solid #EF423C;margin:0 0 24px 0;" color="#EF423C" size="3" noshade>

  <p style="margin:0 0 16px 0;">
    {{body content}}
  </p>

  <p style="margin:16px 0 0 0;">
    {{closing}}
  </p>

</div>
```

### Horizontal Rule Notes

The `<hr>` tag uses both CSS and HTML attributes for cross-client compatibility:
- `style="border:none;border-top:3px solid #EF423C;margin:0 0 24px 0;"` - for modern clients (Gmail, Apple Mail)
- `color="#EF423C" size="3" noshade` - HTML attributes for Outlook's Word rendering engine

### List Formatting

Lists use simple line breaks with dash prefixes. Do NOT use `<ul>/<li>` (inconsistent rendering across email clients).

```html
<div style="margin:0 0 16px 0;">
  - First item<br>
  - Second item<br>
  - Third item
</div>
```

### Link Formatting

```html
<a href="{{url}}" style="color:#5481C9;text-decoration:none;">{{text}}</a>
```

## 5. Signature

Rock Agile signatures are generated by `/ra-email` from a **hosted template** plus a **per-user local config**.

### Architecture

- **Hosted template** lives in this repo at `signatures/template.html` (no-tagline, spacious spacing) and `signatures/template-tagline.html` (with "Back-Office & ERP Specialists" tagline, tighter spacing). One canonical design, easy to update for everyone at once.
- **Per-user local config** lives at `~/.claude/rock-agile-signature.json` on each user's machine. Holds the user's name, email, phone, website, default title, available titles, and tagline-by-default preference. Never goes in git.
- **Per-user override** at `signatures/{email-prefix}.html` in this repo is the escape hatch for anyone who wants a fully custom signature design. Not used by default.

### Rendering rules

- The signature is a `<table>` (the one allowed table in the email standard - see Design Principles).
- The rook image is hosted at `https://raw.githubusercontent.com/RockAgile/rock-agile-tools/main/assets/rook-red.png`. Reference by URL, not inline.
- Phone number link uses `tel:` with the E.164 format (e.g., `tel:+12147364971`). The displayed phone uses the human format (`(214) 736-4971`).
- Title placeholder is uppercased.
- The signature replaces the typed "Thanks, [Name]" sign-off. The email body ends with "Thanks," followed by the signature block. No name on the "Thanks" line - the signature has it.

### Onboarding new users

First-run flow: when `/ra-email` runs and `~/.claude/rock-agile-signature.json` does not exist, prompt the user inline for name/email/phone/website/default-title/other-titles/tagline-preference. Save to the config. No git, no PR, no `gh` CLI required.

Existing Gmail signature: each user should disable their Gmail auto-appended signature so it doesn't double up with the one `/ra-email` embeds.

## 6. Gmail Draft API Notes

When creating a draft via the Gmail API:

- Use the `htmlBody` field for HTML content
- Gmail will strip `<style>` blocks - all styles must be inline
- Gmail will strip `<script>` tags
- Gmail preserves inline styles on most elements
- The signature is embedded in the HTML body by `/ra-email`. Users should disable Gmail's auto-signature for the same account.
- If no recipient email is provided, use `placeholder-{name}@rockagile.io` (e.g., `placeholder-ian@rockagile.io`). If no name is available, use `placeholder-recipient@rockagile.io`. These bounce safely on Rock Agile's servers if accidentally sent.
